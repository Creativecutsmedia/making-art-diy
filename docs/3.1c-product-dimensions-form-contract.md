# 3.1c Product Dimensions Form — kontrakt

**Status:** Kontrakt — låses INDEN kode skrives. W3 step 3.1c, 2026-04-27.
**Formål:** Definér admin-v2 produkt-edit-form's tilføjelse af vægt + dimensioner felter. Whitelist-validation i admin-products-write — `Number.isInteger` + ≥ 0 (post-fix #5).
**Output:** Reference-doc. Ingen kode i denne PR.

**Læs først:** `docs/3.1a-admin-products-write-contract.md` (passthrough §76-78, ingen field-deletion §56), `docs/3.1b-product-edit-form-contract.md` (form-pattern, dirty-tracking caveat §149), `admin-v2/src/hooks.jsx`, `admin-v2/src/page_product_edit.jsx`.

## TL;DR

3.1c udvider admin-v2 produkt-edit-form med 4 nye felter under ny "Leveringsdetaljer"-sektion: `weight_grams` (int gram), `length_cm` / `width_cm` / `height_cm` (int cm). Backend whitelister felterne (`Number.isInteger` + ≥ 0) per fix #5; oprindeligt designet som passthrough per 3.1a §76-78 men HTML5 `min=0` håndhæves ikke uden form-submit. Felterne er offentligt synlige (ingen `INTERNAL_KEYS`-ændring i `update-products.py`). Alle felter er optional på alle produkter — eksisterende produkter behøver ikke migreres, felterne udfyldes editor-driven.

3.1c er **edit-only**. Create + delete = W5. Cloudinary upload = W4. Levering-tids enum (`delivery_time`) = W7-W10 ordrer/forsendelse.

## Scope

### In-scope (3.1c)

- 4 nye felter i form-state: `weight_grams`, `length_cm`, `width_cm`, `height_cm` (alle int, optional)
- Ny "Leveringsdetaljer"-form-section mellem Description og Internal Notes
- Layout: form-grid-2 main, nested form-grid-3 for L/W/H
- Read-adapter (`remapProduct`) udvides med 4 nye type-checks
- Write-adapter (`formToFields`) udvides med 4 conditional spreads (blank → omit)
- HTML5 input-attrs (`type=number`, `min=0`, `step=1`) + frontend pre-Save JS-guard mod negative tal (per fix #5; HTML5 `min=0` håndhæves ikke uden form-submit)
- 4 nye i18n-keys (DA + EN); drop af 2 mockup-rest keys (`weight_g`, `dimensions`)
- Throwaway-test-produkt (`MAD-TEST-997`) for browser-flow

### Out-of-scope

- `delivery_time` enum (`fast`/`med`/`slow`) — W7-W10 ordrer/forsendelse-domæne; i18n-keys parkeres ubrugte til den fase
- Field-clearing via blank-input — 3.1a-limitation (ingen field-deletion); workaround: sæt til andet tal
- Decimal-præcision (`step=0.1`) — post-launch hvis behov; pre-launch er int cm tilstrækkeligt
- Backend max-cap-validation (fx weight ≤ 100kg, length ≤ 200cm) — accepteret tradeoff for solo-admin trust; whitelist tjekker kun `Number.isInteger` + ≥ 0 (per §150)
- 4. produkt-kategori "Værksted/Garage/Skur" — parkeret post-launch v1.1 (søndags-review 2026-04-26)
- Migration-script for eksisterende produkter — editor-driven; Malik tilføjer felter når han næste gang editor produktet
- Per-felt validation_failed-rendering for de nye felter — banner-only design (se §150 + §152); `details.fields.weight_grams` rammer kun bundlet med ikke-dim-fejl
- Deep-equality dirty-tracking — alle nye felter er primitives; eksisterende shallow JSON-equality (3.1b) virker uden ændring

## Integration: form ↔ 3.1a passthrough

POST-payload udvides via `formToFields` med (op til) 4 nye keys i `fields`-objekt. Endpoint-ændring (per fix #5): `validateMerged()` whitelist-check for de 4 dim-felter, bagudkompatibel.

```http
POST /.netlify/functions/admin-products-write
Content-Type: application/json
Cookie: nf_jwt=<...>

{
  "slug": "3d-printet-vase",
  "sku": "MAD-VASE-001",
  "fields": {
    "title": "...",
    "price": 79,
    "weight_grams": 500,
    "length_cm": 20,
    "width_cm": 15,
    "height_cm": 10
  }
}
```

3.1a §76-78 passthrough garanterer:

- Felter ikke i whitelist accepteres uden validering
- Type bevares (int → YAML-int, ikke string)
- Frontend-frihed: vi kan tilføje felterne uden backend-PR

**3.1c-undtagelse (post-fix #5):** `weight_grams` + `length_cm` + `width_cm` + `height_cm` er nu i whitelist (`Number.isInteger` + ≥ 0). 3.1a passthrough-garanti gælder kun for andre fremtidige felter.

## Field-mapping table

| Form-state | Adapter-output (passthrough) | Type | Coercion |
|---|---|---|---|
| `form.weight_grams` (string) | `fields.weight_grams` (int) | int | `parseInt(form.weight_grams, 10)` hvis non-blank, ellers OMIT |
| `form.length_cm` (string) | `fields.length_cm` (int) | int | `parseInt(form.length_cm, 10)` hvis non-blank, ellers OMIT |
| `form.width_cm` (string) | `fields.width_cm` (int) | int | `parseInt(form.width_cm, 10)` hvis non-blank, ellers OMIT |
| `form.height_cm` (string) | `fields.height_cm` (int) | int | `parseInt(form.height_cm, 10)` hvis non-blank, ellers OMIT |

Pattern matcher eksisterende `parseInt(form.price, 10)` i `formToFields` (`hooks.jsx:43`).

### Spread-conditional pattern (blank-omit)

```js
function formToFields(form) {
  return {
    title: form.name_da,
    // ... eksisterende felter ...
    internal_notes: form.notes,
    ...(form.weight_grams !== '' && { weight_grams: parseInt(form.weight_grams, 10) }),
    ...(form.length_cm    !== '' && { length_cm:    parseInt(form.length_cm, 10) }),
    ...(form.width_cm     !== '' && { width_cm:     parseInt(form.width_cm, 10) }),
    ...(form.height_cm    !== '' && { height_cm:    parseInt(form.height_cm, 10) }),
  };
}
```

**Hvorfor OMIT (ikke null):** hvis vi sendte `weight_grams: null` for blank, ville backend's shallow-merge nulstille evt. eksisterende værdi i frontmatter ved enhver redigering hvor brugeren ikke rører feltet. Eksempel-bug: bruger har `weight_grams: 500`, ændrer kun price, klikker Save → weight_grams blive nullet. OMIT-pattern: blank = "ingen ændring i denne edit" (matcher 3.1a partial-update semantik).

### Caveat: ingen field-clearing via blank-input

Bruger kan IKKE clear et eksisterende værdi via blank-input i 3.1c. Workaround: skift til andet tal (fx 0). Ægte "remove-this-key"-funktionalitet kræver `unset_fields[]`-array i 3.1a-kontrakt — post-launch hvis behov.

## Validation-regler

| Felt | HTML5 input-attrs | Pre-POST coercion | Backend-validation |
|---|---|---|---|
| `weight_grams` | `type=number, min=0, step=1` | `parseInt` | `Number.isInteger` + ≥ 0 |
| `length_cm` | `type=number, min=0, step=1` | `parseInt` | `Number.isInteger` + ≥ 0 |
| `width_cm` | `type=number, min=0, step=1` | `parseInt` | `Number.isInteger` + ≥ 0 |
| `height_cm` | `type=number, min=0, step=1` | `parseInt` | `Number.isInteger` + ≥ 0 |

**Pre-POST validation:** HTML5 type-restriction (decimal-input afvises eller trunkeres af `parseInt`) + JS pre-Save guard mod negative tal (per fix #5). HTML5 `min=0` styler kun feltet (`:invalid` CSS) men håndhæves ikke uden form-submit; JS-guard fanger -10 før fetch.

**Backend-validation:** whitelist — `weight_grams` + `length_cm` + `width_cm` + `height_cm` skal være `Number.isInteger` og ≥ 0. Backend returnerer `error: 'invalid_dimension'` (banner-only) hvis kun dim-felter fejler, ellers `validation_failed`. Belt-and-suspenders med frontend-guard fordi HTML5 min=0 ikke håndhæves uden form-submit. Solo-admin pre-launch tradeoff for max-cap: ingen — admin trust.

**Per-felt fieldErrors:** SKIP. Backend mapper pure dim-fejl til `errorKey: 'invalid_dimension'` (banner-only); kun bundlede fejl (dim + ikke-dim sammen) bruger `validation_failed` med `details.fields`. Selv da har dim-felter ingen JSX-rendering for `fieldErrors[X]` — bevidst, banner-only design. Tilføj keys til `backendToForm`-map + rendering hvis behov opstår.

## Forward-compat

3.1c forbereder for:

- **CoolRunner halv-auto integration (W7-W10):** `weight_grams` + `dim_*_cm` felter er præcis hvad fragt-API'er kræver for at beregne pakke-størrelse + porto. Ingen format-konvertering nødvendigt.
- **Per-felt rendering hvis behov:** `backendToForm`-map (`page_product_edit.jsx:110`) kan udvides med dim-keys + 4 `fieldErrors`-rendering linjer i JSX hvis banner-only viser sig utilstrækkelig.
- **Decimal-præcision:** hvis post-launch behov for `step=0.1` cm: skift `parseInt` til `parseFloat` i `formToFields` + opdater HTML5 step-attr. Ingen frontmatter-migration nødvendig (YAML int → float er forward-compat).
- **Internal-only flag:** hvis weight/dim post-launch flyttes til admin-only, tilføj nøgler til `INTERNAL_KEYS` i `update-products.py:12`. Frontend uændret.
- **Manuel ordre-oprettelse (W7-W10, MUST per LAUNCH-CHECKLIST 2026-04-27):** weight/dim på produkt-niveau muliggør automatisk pakke-størrelse-beregning ved manuel ordre, så Malik ikke skal indtaste pr. ordre.

## Test plan (Phase 4 — manuel browser-flow på deploy-preview)

### Throwaway-strategi

Mini-PR FØR feature-PR (PR B i sekvens):

```yaml
# _products/test-3-1c-throwaway.md
---
title: "⚠️ TEST 3.1c — DO NOT BUY (will be deleted)"
title_en: "⚠️ TEST 3.1c — DO NOT BUY (will be deleted)"
sku: MAD-TEST-997
price: 99999
category: Voksne
description: "Test produkt for 3.1c (vægt + dimensioner) — slettes efter merge"
description_en: "Test product for 3.1c (weight + dimensions) — deleted after merge"
published: true
---
```

`published: true` påkrævet — `update-products.py:49` filtrerer published:false væk fra products.json så admin-v2 ikke kan finde produktet til test.

`price: 99999` + advarsels-title beskytter mod utilsigtet køb hvis nogen browser shopen mens PR B er live på master.

`MAD-TEST-997` undgår kollision med 3.1b's `MAD-TEST-998` (slettet, men SKU bevares i `git log`).

Cleanup-commit i PR C sletter throwaway samtidig med feature-merge.
