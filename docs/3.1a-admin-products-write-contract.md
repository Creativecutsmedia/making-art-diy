## TL;DR

POST-endpoint `/.netlify/functions/admin-products-write` modtager en partial edit til ét eksisterende produkt og committer ændringen til `_products/{slug}.md` på GitHub via Contents API. Backend tjekker admin-rolle, fetch'er eksisterende fil, merger ændringer, valider merged resultat mod whitelist, skriver tilbage med `gray-matter`. Last-write-wins (ingen optimistic locking i 3.1a). Customer-facing pages opdateres efter Netlify rebuild (~30-60s eventual consistency).

3.1a er **edit-only**. SKU er immutable identifier — kan ikke ændres ved edit (det ville være delete+create, som er W5). Create + delete + publish-toggle = W5. Cloudinary upload = W4. Vægt/dimensioner = 3.1c (forward-compat via passthrough — ingen schema-ændring krævet i 3.1a).

| | |

## Request

```json
{
  "slug": "3d-printet-vase",
  "sku": "MAD-VASE-001",
  "fields": {
    "price": 79,
    "description": "Opdateret beskrivelse"
  }
}
```

### Top-niveau felter

- **`slug`** (string, required) — filnavn uden `.md`. Backend bruger som GitHub-path: `_products/{slug}.md`.
- **`sku`** (string, required) — sanity-check at `_products/{slug}.md`'s frontmatter-SKU matcher. Hvis mismatch → 409.
- **`fields`** (object, required) — kun ændrede felter. Backend merger på eksisterende frontmatter. Felter ikke nævnt = uændrede.

### Update-model: partial-update + merged-validation

Backend henter eksisterende frontmatter, merger `fields` ovenpå (shallow merge), validerer det MERGED resultat. Det betyder:

- Hvis client kun sender `price`: kun price ændres, øvrige felter bevares som de er
- Validering kører på merged objekt: hvis eksisterende `title` er non-empty og request ikke ændrer den, validation passerer
- **Field-deletion er IKKE understøttet i 3.1a.** Sending `null` eller `""` er semantically "set to that value", ikke "delete key". Hvis post-launch behov for sletning, tilføj eksplicit `unset_fields: [...]` array.

### `sku` er identifier — ikke editable

Backend skriver aldrig request `sku` til frontmatter. Selv hvis `fields.sku` er sendt, ignoreres den (eller afvises med 400). SKU er immutable identifier i 3.1a.

### Whitelist-validering (på merged resultat)

| Felt | Required (merged) | Valideres som |
|---|---|---|
| `title` | ✓ | non-empty string |
| `price` | ✓ | int >= 0 |
| `category` | ✓ | enum: `Børn` \| `Voksne` \| `Erhverv` |
| `description` | ✓ | non-empty string |
| `title_en` | optional | string hvis present |
| `description_en` | optional | string hvis present |
| `image` | optional | string hvis present |
| `extra_images` | optional | list-of-strings hvis present |
| `published` | optional | bool hvis present |
| `internal_notes` | optional | string hvis present |
| Øvrige felter | passthrough | ingen validering, type-bevares |

**Passthrough-regel:** ukendte felter accepteres uden validering og skrives uændrede til frontmatter. Forward-compat for 3.1c (vægt, dimensioner) uden backend-ændring.

## Response

### Success (200)

```json
{
  "ok": true,
  "slug": "3d-printet-vase",
  "sku": "MAD-VASE-001",
  "new_sha": "a1b2c3d4e5f6...",
  "build_eta_seconds": 60
}
```

- **`new_sha`** — GitHub-returneret SHA for nye `_products/{slug}.md`. Frontend kan caches hvis vi senere vil have optimistic locking.
- **`build_eta_seconds`** — fast 60 sekunder (konservativt; faktisk typisk 30-50s). Underpromise + overdeliver. Frontend bruger i success-toast: "Gemt — kunder ser ændringen om ca. 1 minut".

### Fejl-shape (alle 4xx/5xx)

```json
{
  "ok": false,
  "error": "<error_code>",
  "details": { /* feltspecifikke fejl, kun ved validation_failed */ }
}
```

### Fejl-koder

| HTTP | `error` | Trigger |
|---|---|---|
| 400 | `invalid_request` | JSON-parse fejl, manglende `slug`/`sku`/`fields` på top-niveau, eller `fields.sku` sendt |
| 400 | `validation_failed` | Whitelist-validation fejler på merged resultat. `details.fields` indeholder feltspecifikke fejl |
| 401 | `unauthorized` | `event.clientContext.user` mangler |
| 403 | `forbidden` | User mangler `admin`-role |
| 404 | `product_not_found` | `_products/{slug}.md` findes ikke på GitHub |
| 409 | `sku_mismatch` | request `sku` matcher ikke frontmatter-SKU i eksisterende fil |
| 409 | `concurrent_write` | GitHub returnerede SHA-conflict (sjælden race; frontend kan retry) |
| 429 | `rate_limited` | GitHub rate limit (5000/time pr. PAT — ekstremt usandsynligt) |
| 500 | `internal_error` | Uventet exception |
| 502 | `github_error` | GitHub returnerede 5xx eller uventet response-shape |

### `details` ved `validation_failed`

```json
{
  "ok": false,
  "error": "validation_failed",
  "details": {
    "fields": {
      "price": "must be a non-negative integer",
      "category": "must be one of: Børn, Voksne, Erhverv"
    }
  }
}
```

### `details` ved `sku_mismatch`

```json
{
  "ok": false,
  "error": "sku_mismatch",
  "details": {
    "expected": "MAD-VASE-001",
    "received": "MAD-WRONG-999"
  }
}
```

## Persistence flow

1. Modtag POST. Parse JSON-body. Verificér top-niveau (`slug`, `sku`, `fields` present; ingen `fields.sku`).
2. Auth: `event.clientContext.user` exists → ellers 401.
3. Authz: `user.app_metadata.roles.includes('admin')` → ellers 403.
4. GET `_products/{slug}.md` fra GitHub Contents API. Hvis 404 → response 404.
5. Decode base64 content. Parse med `gray-matter` → `{ data: frontmatter, content: body }`.
6. Sanity-check: `data.sku === request.sku` → ellers 409 `sku_mismatch`.
7. Shallow-merge `request.fields` ovenpå `data` (excluding any `sku` key i fields).
8. Validér merged `data` mod whitelist. Fejl → 400 `validation_failed` med field-details.
9. Re-stringify: `matter.stringify(content, mergedData).replace(/\n+$/, '\n')`.
10. Encode base64. PUT til GitHub Contents API:
    - `path`: `_products/{slug}.md`
    - `content`: base64
    - `sha`: SHA fra step 4
    - `message`: `admin: update product {SKU} via admin-v2`
    - `branch`: master (eller default)
11. Hvis GitHub 200/201 → response 200 med `new_sha`.
    - Hvis GitHub 409 → response 409 `concurrent_write` (ingen backend-retry).
    - Hvis GitHub 5xx → response 502 `github_error`.
    - Hvis GitHub 429 → response 429 `rate_limited`.
12. Netlify auto-trigger build (commit til master). Customer-facing pages opdateres efter ~30-60s.

## gray-matter format-normalisering caveat

`gray-matter` parser inputs i forskellige YAML-stilarter (double-quoted, single-quoted, folded scalars, block-scalars `|-`) men stringifier til kanonisk form:
- Double-quoted strenge → single-quoted hvor muligt
- Folded long-strings → `>-` block-scalars
- Block-scalars `|-` bevares som-er

Build-pipeline (`update-products.py` med `yaml.safe_load`) parser begge former identisk → **ingen funktionel ændring**, build bryder ikke.

**Diff-konsekvens:** ved første edit af en fil med "loose" YAML (pr. 2026-04-26: kun `_products/3d-printet-vase.md`) viser PR-diff'en format-normalisering blandet med faktiske content-ændringer. Efter merge er filen normaliseret; fremtidige edits viser rene diffs.

Accepteret tradeoff for at undgå hand-roll YAML-parser. Alternativ ("migrér alle filer til normaliseret format som separat chore først") afvist — ikke værd for solo hobby-skala.

## Trailing-newline normalisering

`gray-matter.stringify` outputter med 2 trailing newlines (`---\n\n`). Eksisterende filer har 1 (`---\n`). Strip post-stringify:

```js
const output = matter.stringify(content, data).replace(/\n+$/, '\n');
```

## Concurrency

**Last-write-wins.** Backend henter aktuel SHA fra GitHub umiddelbart før PUT. Race-vindue ~700ms (latency mellem GET og PUT).

**Pre-launch acceptabelt:** kun Malik er admin. Realistisk konflikt = 0.

**Hvis GitHub returnerer 409 ved PUT** (SHA blev forældet i race-vinduet): backend retry'er IKKE automatisk i 3.1a — returnerer 409 `concurrent_write` til frontend. Frontend kan vælge at re-fetche produkt + retry.

**Post-launch v1.1 path:** request kan tilføje `expected_sha`-felt; hvis present, sammenlign mod aktuel SHA før PUT og afvis 409 ved mismatch. Backward-kompatibel — clients der ikke sender `expected_sha` får last-write-wins.

## Auth + authorization

```js
const user = event.clientContext && event.clientContext.user;
if (!user) return jsonResponse(401, { ok: false, error: 'unauthorized' });

const roles = (user.app_metadata && user.app_metadata.roles) || [];
if (!roles.includes('admin')) {
  return jsonResponse(403, { ok: false, error: 'forbidden' });
}
```

**Afvigelse fra `admin-stats.js`-pattern:** admin-stats tjekker kun user-presence, ikke role. 3.1a tilføjer eksplicit role-check fordi:

- 3.1a er write-endpoint med side-effects (git commit, build trigger)
- Edge-guard beskytter kun `/admin-v2/*`, IKKE `/.netlify/functions/admin-*`
- Functions er offentligt URL-addressable for enhver Identity-bruger uden role-check

Admin-stats role-hul tracket separat (ikke 3.1a-scope) — bør fixes pre-launch.

Eksplicit IKKE i denne endpoint:

- **Delete product** — W5
